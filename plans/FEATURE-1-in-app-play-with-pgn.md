# FEATURE-1: In-App Play Mode with Live PGN and On-Demand Analysis

## Objective

Replace the current "paste FEN/PGN, then analyze" workflow with a live in-app chess game.
The user should be able to play directly on the board against a lightweight computer opponent, see the PGN update as moves are made, and click `Analyze` whenever they want coaching for the current position.

---

## User Experience

### Current flow

- Play a game elsewhere.
- Paste a FEN or PGN into the app.
- Click `Analyze`.
- View the engine result and coaching hint for that static position.

### Target flow

- Start a new game inside the app.
- Choose to play as White or Black.
- Make moves directly on the board.
- Let a lightweight computer opponent respond automatically.
- See the PGN update live as the game progresses.
- Click `Analyze` at any moment to evaluate the current position and request a coaching hint.

The key behavior change is that analysis becomes **on-demand** instead of being the only way to move the board forward.

---

## Product Requirements

### Must have

- Interactive board inside the app.
- User can play against a simple local engine bot.
- PGN is visible and updates after each move.
- `Analyze` evaluates the **current live position**.
- Coach hint and engine panel still work.
- Analysis is cleared or marked stale after the board changes.
- Starting a new game resets board state, PGN, and prior analysis.

### Nice to keep from current app

- Existing board/eval/coach dashboard layout.
- Existing Stockfish worker for analysis.
- Existing Claude coaching integration.

### Explicit non-goals for this iteration

- Multiplayer.
- Clocks.
- Opening book.
- Difficulty ladder.
- Move takeback.
- Game persistence.
- Rich PGN import/export workflow.

## Proposed Architecture

### 1. Separate live game state from analysis state

The current `useChessAnalysis` hook owns `fen`, which makes sense for a paste-based analyzer but not for an interactive game.
This feature should split responsibilities:

- **Game state hook** owns the actual game (`Chess` instance, move history, current FEN, PGN, player color, turn, game status, bot thinking state).
- **Analysis hook** analyzes the current FEN only when `Analyze` is clicked.

This keeps board play responsive and prevents analysis state from controlling the board.

### 2. Use a dedicated engine instance for bot play

The app already uses a Stockfish web worker for analysis.
Bot play should use a **separate** engine instance to avoid collisions between:

- long-running analysis requests
- automatic bot move generation

The bot engine can run at a lower depth or short movetime so it stays lightweight.

### 3. Keep engine display White-centric, but coach hints player-centric

The current engine panel and eval bar are standard chess UI elements and should continue to show evaluation from White's perspective.
The coach hint should be framed from the human player's perspective, which is especially important when the user chooses to play Black.

---

## UI Changes

### Replace the text input bar with game controls

The current `InputBar` is centered around pasted FEN/PGN input.
For live play, it should be replaced by controls such as:

- `New Game`
- `Play as White`
- `Play as Black`
- `Analyze`

### Add a live PGN panel

The dashboard should display the PGN generated by the current game.
This can be shown as a scrollable read-only panel or move list.

### Add status text

The interface should surface:

- whose turn it is
- whether the bot is thinking
- whether the game is over
- the result if applicable

## Key Gotchas

### 1. Do not use one engine instance for both bot moves and analysis

The UCI worker is command-driven and asynchronous, so mixing automatic bot commands with manual analysis on one worker can produce race conditions and incorrect listeners.

### 2. PGN requires full move history, not just FEN

FEN alone is not enough to reconstruct the live PGN string, so the live game must preserve move history via a `Chess` instance or equivalent history-aware state.

### 3. Analysis must not mutate board state

The board position should always come from the live game state, and analysis should be a read-only action over the current FEN.

### 4. Coaching perspective should follow the human player

Once the user can choose White or Black, coaching should address the human player's side rather than blindly assuming the side to move is the player.

### 5. Clear stale analysis after moves

If the user makes another move after analyzing, the UI should not silently present old engine lines as if they belong to the new position.
